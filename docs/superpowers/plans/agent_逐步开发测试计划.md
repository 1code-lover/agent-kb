# Agent 逐步开发测试计划

## 1. 使用原则

本计划用于指导桌面端 Agent 的逐步开发与逐步放行。

推进规则只有一条：

1. 完成当前阶段开发
2. 执行当前阶段测试
3. 回归上一阶段关键能力
4. 测试全部通过
5. 才允许进入下一阶段

如果当前阶段测试未通过，不允许继续叠加新功能。

---

## 2. 总体目标

当前阶段先不急着扩知识库，而是先把 Agent 本体做成一个可运行、可配置、可切换、可审批、可审计的桌面产品基础版。

本轮优先级顺序：

1. 模型配置可用
2. Agent 纯模型路径可用
3. 工具能力可用
4. 审批闭环可用
5. 状态与回执稳定
6. 最后再接知识库

---

## 3. 阶段划分

### 阶段 1：模型配置基础链

#### 开发目标

- 支持自定义 provider
- 支持自定义 `base_url`
- 支持自定义 `api_key`
- 支持保存模型列表
- 支持读取当前模型配置

#### 开发项

- 后端：
  - `api/services/model_service.py`
  - `api/routers/settings.py`
- 前端：
  - `webapp/src/pages/ModelsPage.jsx`
  - `webapp/src/api/models.js`

#### 测试项

- 能新增一个自定义 provider
- `GET /api/model/options` 能看到新增 provider
- 能保存当前模型配置
- 再次读取 options 时能看到 `current_llm_info`
- 能删除 provider
- 删除后 provider 从列表消失

#### 放行标准

- 增删改查全通
- 前端页面可操作
- 前端 build 通过

#### 失败处理

- 如果 provider 保存失败，先修后端 contract
- 如果前端显示不一致，先修前端状态映射
- 未通过不得进入阶段 2

---

### 阶段 2：模型连接测试链

#### 开发目标

- 支持模型连通性测试
- 支持返回明确成功或失败信息

#### 开发项

- 后端：
  - provider test 接口
  - 错误分类
- 前端：
  - 测试按钮
  - 结果展示

#### 测试项

- 正确 `base_url/api_key/model` 能返回可连接
- 错误 `base_url` 返回连接失败
- 错误 `api_key` 返回认证失败或拒绝
- 错误 `model` 返回模型不可用
- 页面能清晰显示错误，而不是静默失败

#### 放行标准

- 测试结果可靠
- 错误信息可读
- 不影响阶段 1

#### 失败处理

- 若错误信息不可用，优先修测试接口返回结构
- 若页面提示不清楚，优先修前端错误展示

---

### 阶段 3：Agent 纯模型路径

#### 开发目标

- `mode=agent` 直接使用当前选中的模型配置
- 不依赖知识库也能工作
- 能完成普通问答与基础任务拆解

#### 开发项

- 后端：
  - `agent_router` 路由到当前模型调用路径
  - `agent_tools` 新增 `llm_chat`
  - 统一配置读取
- 前端：
  - Agent 首页显示当前模型

#### 测试项

- 在模型页先保存一个当前模型
- Agent 模式提交普通问题
- receipt 中能看到当前 `provider/model`
- timeline 正常返回
- 没装知识库依赖时，Agent 模式仍能运行

#### 放行标准

- `agent` 模式可用
- 模型切换后，下一轮运行使用新模型
- 不再错误走 `kb_search`

#### 失败处理

- 若仍走错路由，先修 router
- 若模型配置读不到，先修共享 store
- 若远端 provider 报错但本地链路正确，可记录为“配置正确，远端不可用”

---

### 阶段 4：Agent 快速切模

#### 开发目标

- 不离开 Agent 页面即可切模型
- 切换后下一轮运行立即生效

#### 开发项

- Agent 首页增加当前模型展示
- 增加 quick switch
- 复用 `model/select`

#### 测试项

- 模型页保存一个 provider 和多个模型
- Agent 页能显示当前模型
- Agent 页能快速切到另一个模型
- 下一次提交任务时 receipt 中模型已变化
- 刷新页面后仍保持最新模型

#### 放行标准

- 不离开 Agent 页即可切模
- 切模后下一轮任务立即生效

#### 失败处理

- 若 Agent 页切换成功但运行未生效，优先检查运行时缓存 fingerprint

---

### 阶段 5：`read_file` 工具

#### 开发目标

- Agent 能读取本地文件
- 返回摘要与回执

#### 开发项

- `read_file` 工具
- 输入提示
- 路径模板

#### 测试项

- 输入真实存在路径，返回摘要
- 输入不存在路径，返回明确错误
- 中文文件可读
- 英文文件可读
- receipt 中包含路径和字符数

#### 放行标准

- `read_file` 稳定
- 错误路径不会打挂页面

#### 失败处理

- 路径解析问题先修 router/path extract
- 文件读取问题先修工具层

---

### 阶段 6：`run_cmd` 低风险链

#### 开发目标

- 低风险命令可直接执行
- 返回标准化回执与状态

#### 开发项

- 命令 allowlist
- 回执与 timeline 显示

#### 测试项

- `whoami` 可运行
- `pwd / dir / ls` 可运行
- 返回 `exit_code/output`
- 命令失败时状态正确
- 页面 timeline 与 receipts 同步

#### 放行标准

- 低风险命令链稳定
- 状态与回执一致

#### 失败处理

- 若命令执行正常但页面状态不对，优先修 hydrate
- 若命令越权，优先修 allowlist

---

### 阶段 7：审批链

#### 开发目标

- 高风险命令进入审批
- 批准或拒绝后形成闭环

#### 开发项

- 风险分级
- pending action 队列
- 审批面板
- 审批后的执行回流

#### 测试项

- 高风险命令进入 `pending_actions`
- 页面能看到待审批动作
- 拒绝后 task_state 结束，timeline 增加拒绝结果
- 批准后命令真正执行
- receipts 至少包含：
  - pending receipt
  - approval receipt
  - run receipt
- pending_actions 被清空

#### 放行标准

- 审批链完整
- 不重复执行
- 不残留 pending 状态

#### 失败处理

- 若批准后未执行，先修 approval -> execute 链
- 若执行了但没回流，先修 approval 返回结构

---

### 阶段 8：稳定性与回归

#### 开发目标

- 前 1 到 7 阶段可重复执行
- 多轮操作不串状态

#### 开发项

- session 隔离
- 重复点击保护
- 状态恢复
- 错误提示
- 日志/receipt 补齐

#### 测试项

- 连续多轮运行不串 session
- 连续切模型不丢状态
- 审批按钮重复点击不重复执行
- API 重启后前端不崩
- 桌面端启动链仍命中 `127.0.0.1:18080/api/health`

#### 放行标准

- 多轮使用稳定
- build 通过
- 核心 API 回归通过

#### 失败处理

- 稳定性问题优先于新功能
- 未通过不得进入知识库阶段

---

## 4. 每阶段固定执行流程

每一阶段都必须按下面顺序执行：

1. 明确本阶段目标
2. 只开发本阶段所需能力
3. 执行本阶段测试
4. 回归上一阶段关键能力
5. 记录测试结果
6. 确认放行
7. 再进入下一阶段

## 6. 当前建议执行位置

结合当前项目状态，建议从下面位置开始正式按计划推进：

- 阶段 1：已基本完成，可补测试记录
- 阶段 2：已基本完成，可补测试记录
