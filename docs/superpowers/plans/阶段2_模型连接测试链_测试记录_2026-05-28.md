# 阶段 2 测试记录

- 阶段名称：模型连接测试链
- 开发目标：
  - 支持模型连通性测试
  - 支持返回明确成功或失败信息
- 开发完成时间：2026-05-28 16:10 +08:00

## 测试环境

- 本地 API：`http://127.0.0.1:18080`
- 测试接口：`POST /api/model/providers/test`

## 测试项

- [ ] 正确 `base_url/api_key/model` 能返回可连接
- [x] 错误 `base_url` 返回连接失败
- [x] 不可达地址返回连接失败
- [x] 错误 `api_key` 返回明确失败信息
- [ ] 错误 `model` 返回模型不可用
- [ ] 页面能清晰显示错误，而不是静默失败

## 测试步骤摘要

1. 用 `https://example.com/v1` 作为伪 OpenAI 兼容地址发起测试
2. 用 `http://127.0.0.1:9/v1` 作为不可达地址发起测试
3. 之前已做过一次 OpenAI 兼容地址的错误 key 负向测试，返回 `reachable=false` 和可读 `detail`

## 测试结果

- 通过：
  - 错误 `base_url` 返回 `reachable=false`
  - 不可达地址返回 `reachable=false`
  - 返回体包含可读 `detail`
  - 接口不会静默失败
- 失败：
  - 本阶段尚未完成真实有效 provider 的正向连通验收
  - 本阶段尚未完成“错误 model”场景的正式记录
  - 本阶段尚未在本轮复测中补做前端点击链路验收

## 关键证据

- `https://example.com/v1`
  - 返回 `reachable=false`
  - `detail` 含 `http_405`
- `http://127.0.0.1:9/v1`
  - 返回 `reachable=false`
  - `detail` 含连接拒绝信息
- 既有负向测试记录
  - 对 OpenAI 兼容地址使用无效 key 时，返回 `reachable=false`
  - 错误信息可读，可用于页面提示

## 回归结果

- 上一阶段关键能力是否仍正常：
  - 阶段 1 的 provider 增删改查链路未受影响
  - 当前模型配置读写接口仍正常可用

## 放行结论

- 是否允许进入下一阶段：有条件允许
- 结论说明：
  - 从接口稳定性看，阶段 2 的负向链路已基本满足要求
  - 但若按严格门禁定义，仍缺少一条真实有效 provider 的正向连通验证，以及错误 model 的正式验收记录
  - 因此建议结论写成“可进入阶段 3 联调，但阶段 2 在真实 provider 下仍需补最终放行证据”
