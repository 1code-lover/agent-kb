# ThinkRAG 面试问答

## 一、1分钟项目介绍

`ThinkRAG` 是一个基于 `LlamaIndex + Streamlit` 构建的本地 RAG 问答系统，核心目标是让用户在本地知识库上进行准确问答。用户可以上传文件或者网页内容，系统会完成文档解析、文本切分、向量化、索引构建，然后在提问时通过混合检索和可选重排序找到最相关的知识片段，再交给大模型生成答案。这个项目的特点是对中文场景做了比较多优化，比如中文分词、中文标题增强、中文提示模板，以及对本地模型和国产 API 的兼容支持。

## 二、3分钟项目介绍

这个项目本质上是一个可落地的本地知识库问答系统，不只是简单调用大模型，而是完整实现了一条 RAG 链路。

前端我用的是 `Streamlit`，因为它适合快速做一个可交互的多页面应用。用户可以在页面里配置 LLM、嵌入模型、Reranker，也可以上传文件、抓取网页、管理知识库，然后直接进入问答页面提问。

后端核心是 `LlamaIndex`。文档进入系统后，会先经过 `IngestionPipeline`，包括文本切分、中文标题增强、向量化，然后把结果写入索引。项目里有一个 `IndexManager` 负责索引的创建、加载、插入和删除。

在检索阶段，我没有只用单一向量检索，而是做了混合检索。一方面用向量检索解决语义相似问题，另一方面用 BM25 解决关键词精确匹配问题。因为默认 BM25 对中文支持不好，所以我用 `jieba` 做了中文分词。检索结果再通过 `FusionRetriever` 做融合，必要时再加一个 `Reranker` 做精排，提高答案准确率。

存储方面，这个项目区分开发模式和生产模式。开发模式用本地文件存储，开箱即用，不依赖数据库；生产模式会切到 `Redis + Chroma`，分别承担文档存储、索引存储、聊天记录和向量存储。整体上，这是一个从产品交互、RAG 主链路、中文优化到存储分层都比较完整的项目。

## 三、项目解决了什么问题

它解决的是“用户有一批私有资料，但普通大模型不知道这些资料内容”的问题。

大模型本身只会基于预训练知识回答，无法准确回答用户本地文档里的具体内容。这个项目通过把文档构建成知识库，在提问时先检索相关内容，再基于检索结果生成答案，让模型能围绕本地知识回答问题。

## 四、完整 RAG 流程是什么

可以概括成 6 步：

1. 用户上传文件或者输入网页 URL。
2. 系统读取原始内容，做文本切分。
3. 对切分后的文本块做向量化，并写入索引。
4. 用户提问后，系统先做检索，找出最相关的文本块。
5. 可选地对检索结果做重排序。
6. 最后把问题和上下文一起交给 LLM 生成答案，并返回来源片段。

如果用一句话总结，就是：

**离线知识接入 -> 结构化切块 -> 建索引 -> 混合检索 -> 重排 -> 基于上下文生成回答。**

## 五、为什么选择 LlamaIndex

我选 `LlamaIndex` 主要是因为它非常适合做文档到问答这条链路。它把文档读取、节点切分、索引构建、检索器、查询引擎这些能力都封装得比较完整，我可以把精力放在 RAG 设计和中文优化上，而不是从零拼所有基础设施。

另外它的扩展性也比较好，比如：

- 可以自定义 `Retriever`
- 可以插入 `Reranker`
- 可以替换存储后端
- 可以自定义 `Prompt Template`

所以它适合这个项目从 demo 逐步走向完整应用。

## 六、为什么选择 Streamlit

我选 `Streamlit` 是因为这个项目强调的是“快速搭建一个可交互的本地应用”。它非常适合这种场景：

- 页面开发速度快
- 适合做配置页、上传页、表格管理页
- 和 Python 后端天然结合
- 不需要前后端分离就能快速上线本地版本

当然它也有缺点，比如前端交互复杂度有限、组件能力不如 React 灵活，所以如果以后做产品化桌面端或者企业级前端，我会考虑换成 `React/Vite + FastAPI`。

## 七、为什么要做混合检索，而不是只做向量检索

因为向量检索和 BM25 各有优缺点，单独用一个都不够稳。

- 向量检索擅长语义相似，能找到“意思接近”的内容
- BM25 擅长关键词精确匹配，尤其是专有名词、术语、数字、缩写

如果只用向量检索，可能会漏掉关键词非常明确但语义表达不相近的内容。如果只用 BM25，又容易漏掉“换一种说法”的语义相关内容。

所以我这里用的是混合检索，把两种能力结合起来，整体召回更稳。

## 八、向量检索和 BM25 各自适合什么场景

向量检索更适合：

- 用户问题和原文表达不完全一致
- 需要理解语义相近关系
- 例如“流程有哪些特征”和文档里写的是“流程具有目标导向、重复性和过程性”

BM25 更适合：

- 精确术语查询
- 关键词、缩写、数字、型号
- 比如搜 `glm-4-air`、`Top K`、某个制度名称

所以我的理解是：

**向量检索负责“懂意思”，BM25 负责“抓关键词”。**

## 九、为什么 BM25 在中文场景下要自己做分词

因为默认 BM25 的很多实现是按空格或者英文 token 来切词的，而中文没有天然空格分词。如果直接用默认 tokenizer，中文一整句可能会被错误处理，导致词频统计不准确，检索效果会很差。

所以我在项目里给 BM25 换成了 `jieba` 分词器，这样：

- 文档建索引时能按中文词语切分
- 查询时也能按中文词语切分
- BM25 的词频和逆文档频率计算才有意义

这是这个项目里比较关键的中文优化点之一。

## 十、为什么要做中文标题增强

很多中文文档，尤其是教材、报告、制度文件，结构性很强，标题和正文关系紧密。实际检索时，用户问的问题很可能和标题强相关，但真正命中的文本块是正文块。如果正文块本身没有带标题信息，就会丢失上下文语义。

所以这个项目里做了 `ChineseTitleExtractor`，思路是：

- 如果某个 chunk 被识别成标题
- 就把标题信息注入后续正文块
- 比如变成“下文与（第一章 流程概述）有关。……”

这样做的好处是，即使只检索到正文块，它也携带了章节语义，检索和生成都更稳定。

## 十一、文本切分为什么重要

文本切分是 RAG 里很关键的一步，因为：

- 原始文档太长，不能整篇送给模型
- 向量检索也需要比较细粒度的文本块
- 切得太大，召回不够精准
- 切得太小，上下文信息又容易丢

所以切分本质上是在平衡：

- 检索精度
- 上下文完整性
- 向量表示质量
- LLM 输入成本

## 十二、`chunk_size` 和 `chunk_overlap` 的作用

`chunk_size` 是每个文本块的最大长度。`chunk_overlap` 是相邻文本块之间重叠的部分。

`chunk_size` 太小：

- 信息被切得很碎
- 单块上下文不足
- 回答容易不完整

`chunk_size` 太大：

- 检索粒度变粗
- 噪音增多
- 还会增加 token 成本

`chunk_overlap` 的价值是避免边界截断。比如一句重要信息正好被切在两个块的边界上，如果没有 overlap，两个块都可能信息不完整；有 overlap 后，这段内容至少能完整出现在某个块里。

## 十三、为什么开发模式和生产模式用不同分割器

开发模式更强调简单和零依赖，所以用了 `SentenceSplitter`。生产模式更强调中文分割质量，所以用了基于 `spaCy` 的分割器。

也就是说，这里是一个典型的工程权衡：

- 开发模式优先可运行、易调试
- 生产模式优先精度和效果

## 十四、`IndexManager` 的职责是什么

`IndexManager` 是这个项目里索引生命周期的统一管理者。它主要负责：

- 检查索引是否存在
- 初始化索引
- 加载已有索引
- 插入新的节点
- 从文件构建索引
- 从网页构建索引
- 删除文档及相关节点

我会把它理解成“知识库索引的服务层封装”。这样前端页面不需要直接操作 `LlamaIndex` 的底层对象，而是统一通过 `IndexManager` 处理。

## 十五、为什么 `store_nodes_override=True` 很关键

这个参数很关键，是因为很多向量数据库默认更关注“向量存储”，不一定会把原始节点文本完整保存在 `docstore` 里。

但这个项目不仅要做向量检索，还要做：

- BM25 检索
- 来源展示
- 文档管理和删除

这些能力都依赖能拿到原始 node 文本和 metadata。所以这里强制把节点存进 `docstore`，这样原文不会丢，后续功能才完整。

## 十六、`IngestionPipeline` 做了什么

在这个项目里，摄取流水线主要做了三件事：

1. 文本切分
2. 中文标题增强
3. 向量化

然后把结果写到对应的存储里。另外它用的是 `UPSERTS` 策略，这样重复导入时可以做更新或插入，而不是简单重复堆积数据。

## 十七、为什么要用 `UPSERTS`

因为用户可能会重复上传同一个文件，或者同一份资料更新后重新导入。如果没有 `UPSERTS`，就容易出现重复数据、重复向量、脏索引。

所以 `UPSERTS` 的好处是：

- 已存在就更新
- 不存在就插入
- 更适合知识库的增量维护

## 十八、Reranker 在这个项目里解决了什么问题

Reranker 解决的是“召回到了多个候选，但排序还不够准”的问题。

向量检索和 BM25 更像是“粗排”，先把一批可能相关的内容找出来。Reranker 再拿 query 和候选文档逐个做更精细的相关性判断，重新排序，只保留最相关的几个。

它的价值主要体现在：

- 提高最终答案上下文的质量
- 减少噪音块进入 LLM
- 提升回答准确率

## 十九、为什么 Reranker 不能直接替代 Retriever

因为 Reranker 精度高，但成本大、速度慢。它通常是交叉编码器，要把 query 和每个候选文档成对计算，复杂度高。

如果拿它直接对整个知识库所有文档排序，代价太大。所以正确做法是：

- Retriever 先快速召回一小批候选
- Reranker 再对候选精排

也就是经典的：

**粗排 + 精排**

## 二十、双编码器和交叉编码器怎么解释

双编码器就是 Embedding 检索那一套：

- query 单独编码成向量
- document 单独编码成向量
- 再算相似度

优点是快，适合大规模召回。缺点是 query 和 doc 之间交互不够细。

交叉编码器就是 Reranker：

- query 和 document 一起输入模型
- 直接输出相关性分数

优点是精度高。缺点是慢，不适合全库扫描。

## 二十一、`top_k` 和 `top_n` 的区别

`top_k` 是检索阶段保留多少个候选。`top_n` 是重排序之后最终保留多少个结果。

例如：

- 先检索出 5 个候选，这就是 `top_k = 5`
- Reranker 再从这 5 个里选 2 个最相关的，这就是 `top_n = 2`

## 二十二、为什么嵌入模型不能在知识库已有数据后随便切换

因为不同嵌入模型生成的向量空间不一样，甚至维度都可能不同。旧文档是用老模型编码的，新 query 如果用新模型编码，再去和旧向量比相似度，结果就不可靠了。

所以一旦知识库里已经有数据，切换嵌入模型实际上意味着：

- 旧索引失效
- 需要重新向量化
- 需要重建索引

这也是前端页面里在已有文档时禁用嵌入模型切换的原因。

## 二十三、`state.py` 为什么关键

`state.py` 是整个应用初始化的中枢。它负责把以下内容统一初始化好：

- 当前 LLM
- 当前索引管理器
- 当前嵌入模型
- 当前参数配置
- Ollama 状态
- API 模型选择
- 上传文件和网站列表

我会把它理解成 Streamlit 应用里的“全局状态入口”。没有它的话，多个页面之间状态会很乱，初始化顺序也容易出问题。

## 二十四、`session_state` 和 `CONFIG_STORE` 的区别

`session_state` 是运行时内存状态，适合当前会话中共享。`CONFIG_STORE` 是持久化状态，适合跨重启保留。

比如：

- 当前页面选中的模型、上传列表，这些更偏运行时，用 `session_state`
- 当前 LLM 配置、response_mode、reranker 开关，这些需要重启后保留，用 `CONFIG_STORE`

一句话总结：

- `session_state` 解决“这次运行里怎么共享状态”
- `CONFIG_STORE` 解决“应用重启后配置不丢”

## 二十五、开发模式和生产模式的区别

开发模式主要特点是：

- 零依赖
- 全部用本地文件存储
- 方便本地快速启动和调试

生产模式主要特点是：

- 用 Redis 存文档、索引、聊天记录等
- 用 Chroma 做向量存储
- 更适合稳定运行和数据持久化

这是典型的工程分层思路：

- 开发时先追求低成本、易调试
- 生产时再追求稳定、可扩展

## 二十六、为什么生产模式要引入 Redis 和 Chroma

因为本地 JSON 存储虽然简单，但随着数据变多会有明显问题：

- 读写性能下降
- 管理复杂
- 并发支持弱
- 扩展性差

所以生产模式做了拆分：

- `Redis` 负责结构化存储和快速读写
- `Chroma` 负责向量存储和相似度检索

这样职责更清晰，也更接近真实应用架构。

## 二十七、为什么要保留 docstore、不能只靠 vector store

因为向量库主要存的是向量，核心职责是“找相似”。但系统还需要：

- 读取原文
- 展示来源片段
- 做 BM25
- 管理和删除文档
- 查看 metadata

这些都离不开 `docstore`。所以我的理解是：

- `vector_store` 负责“找到谁”
- `docstore` 负责“这个东西到底是什么”

## 二十八、网页抓取为什么要做双重 fallback

因为网页抓取是不稳定的。不同网站的 HTML 结构差异很大，有些页面还有 JS 动态渲染，所以只靠一种抓取方式很容易失败。

这个项目先用 `BeautifulSoup` 做直接解析。如果抓不到有效正文，就再走 `r.jina.ai` 的镜像方式，相当于让外部服务先把网页整理成更干净的文本或 Markdown 再读。

这样做的意义是提高网页知识接入的鲁棒性。

## 二十九、这个项目里有哪些典型工程化细节

我觉得比较能体现工程意识的点有这些：

1. 中文 BM25 分词替换
2. 中文标题增强
3. 混合检索而不是单检索
4. Reranker 可选开关
5. 开发/生产双模式
6. `CONFIG_STORE` 做配置持久化
7. `store_nodes_override=True` 保证原文可追踪
8. 文件名安全处理，防止路径遍历
9. 网页抓取失败 fallback
10. 流式输出失败时回退到非流式输出

这些都不是“算法炫技”，但很体现可落地性。

## 三十、如果你总结这个项目的亮点，会怎么说

我会总结成 4 点：

1. **链路完整**  
   不是只调一下模型，而是从知识接入、索引构建、检索、重排到回答展示都做全了。

2. **中文优化明确**  
   BM25 中文分词、中文标题增强、中文 prompt，这些都不是纯英文方案平移。

3. **工程化意识比较强**  
   开发/生产模式分离、配置持久化、异常兜底、存储分层这些都考虑到了。

4. **具备扩展空间**  
   后面很容易继续往 FastAPI、多用户、多知识库、前后端分离方向演进。

## 三十一、如果面试官问你“这个项目还有哪些不足”，怎么答

我会比较诚实地说几个点：

1. 前端是 Streamlit，适合快速验证，但复杂交互能力有限。
2. 后端还没有彻底服务化，未来可以拆成 FastAPI。
3. 多用户隔离目前比较弱，当前更偏单机单用户。
4. 网页抓取还不是特别强，复杂站点兼容性可以继续提升。
5. 检索评估体系还可以更系统，比如加入离线评测集和召回指标。

这样回答的好处是既承认不足，又说明你知道演进方向。

## 三十二、如果继续优化，你会优先做什么

我会优先做 3 件事：

1. **做检索评测体系**  
   建立一批标准问答样本，评估不同 `chunk_size`、检索模式、reranker 配置的效果。

2. **后端服务化**  
   把索引、检索、问答接口拆到 `FastAPI`，前端和后端解耦。

3. **支持多用户多知识库**  
   把用户、知识库、索引之间做真正隔离，便于实际使用。

## 三十三、如果问“你在这个项目里最难的问题是什么”，怎么答

我会选“**中文检索质量优化**”来讲。因为最开始如果只是直接套通用 RAG 流程，在中文场景下效果未必稳定。问题主要在两个地方：

- BM25 默认分词不适合中文
- 标题和正文语义断裂，检索经常只命中正文但丢失章节语境

我做的优化是：

- 用 `jieba` 替换 BM25 分词器
- 增加中文标题增强，把标题注入正文 chunk
- 同时用混合检索和可选 reranker 进一步提高结果质量

这个问题比较好，因为它既有技术深度，又能体现你是真的理解了中文 RAG，而不是只会套框架。

我主要负责的是这个项目从知识接入到问答生成这条主链路的理解和实现，包括：

- 文档摄取和索引构建逻辑
- 混合检索和重排序设计
- 中文场景下的文本处理优化
- 模型配置和状态管理
