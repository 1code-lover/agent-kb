# ThinkRAG 项目代码注释强制规范（V1）

## 1. 目的
本规范用于统一项目代码可读性和可维护性，要求研发在编写或修改代码时，为文件、函数和复杂逻辑提供结构化注释。  
该规范为项目级强制要求，适用于新增代码和重构代码。

## 2. 适用范围
- Python 文件（强制）
- JavaScript/TypeScript 文件（建议按同等标准执行）
- 核心逻辑层（server、api、store、agent 编排）必须全量满足

## 3. 强制要求

## 3.1 文件级注释（每个文件开头必须有）
每个源码文件顶部必须有“文件说明注释”，至少包含：
1. 文件功能（做什么）
2. 在系统中的角色（被谁调用，依赖谁）
3. 主执行逻辑（高层流程）

### Python 文件推荐格式
```python
"""
模块功能：
- 负责 xxx

执行逻辑：
1. 接收 xxx 输入
2. 调用 xxx 完成处理
3. 输出 xxx 结果

关键依赖：
- xxx
"""
```

## 3.2 函数级注释（每个函数必须有）
每个函数必须写 docstring，至少包含：
1. 功能说明（函数做什么）
2. 输入参数说明（类型、含义、约束）
3. 执行逻辑说明（核心步骤）
4. 输出说明（返回值结构和语义）
5. 异常/边界说明（如有）

### Python 函数推荐格式
```python
def run_agent(payload: dict) -> dict:
    """
    功能：
    执行一次智能体任务并返回结构化结果。

    输入：
    - payload(dict): 包含 session_id、goal、constraints 等运行参数。

    执行逻辑：
    1. 校验输入参数
    2. 构建执行计划
    3. 调用工具链执行
    4. 汇总结果并返回

    输出：
    - dict: 包含 status、steps、result、error 等字段。

    异常：
    - ValueError: 参数不合法时抛出。
    """
```

## 3.3 复杂逻辑行内注释（必须）
遇到以下场景，必须增加行内注释：
- 分支逻辑复杂（3 个及以上分支）
- 有业务规则映射（如风险等级、策略路由）
- 有性能或兼容性权衡
- 有“看起来反直觉”的实现

要求：
- 注释说明“为什么这样做”，不是重复代码字面含义。
- 注释保持简洁，避免大段无价值描述。

## 3.4 禁止事项
- 禁止“空洞注释”（例如：`# 设置变量`、`# 调用函数`）。
- 禁止注释与代码行为不一致。
- 禁止只注释部分函数，遗漏同文件其他函数。

## 4. 代码评审检查项（必须通过）
1. 文件头注释是否完整。
2. 每个函数是否具备输入/逻辑/输出说明。
3. 复杂逻辑是否有解释“为什么”的注释。
4. 注释是否与当前代码一致。

## 5. 增量落地策略
- 新增文件：100% 满足本规范。
- 修改文件：涉及函数必须补齐注释。
- 历史存量：按“触达即治理”原则逐步补齐。

## 6. 生效规则
本规范自即日起生效。后续所有 Agent 生成代码和人工提交代码都必须遵守本规范；不满足规范的变更不得进入合并流程。

## 7. 自动检查（建议纳入提交流程）
- 检查脚本：`scripts/check_commenting.py`
- 运行方式：
  - `python scripts/check_commenting.py`
  - `python scripts/check_commenting.py server webapp/src`
- 检查范围：
  - Python：模块 docstring + 函数 docstring
  - JS/JSX：文件头注释 + 命名函数紧邻注释
