## 1. 项目定位与技术栈一句话版本

**ThinkRAG** 是一个面向知识问答场景的 RAG 系统，基于 **Streamlit + LlamaIndex** 构建，核心采用 **混合检索（BM25 + 向量）**、**可选 Reranker 重排**、**Ollama/OpenAI 兼容 API**，并支持 **开发/生产双模式存储**，兼顾快速迭代与部署可维护性。

## 4. 高频面试问题与答题要点（12+）

### 1）你为什么要做混合检索，而不是只用向量检索？
- 向量检索擅长语义相似，但对精确关键词、缩写、数字、专有名词不稳定。
- BM25 对关键词命中更强，混合后召回覆盖面更稳。
- 答题重点：说明“问题类型差异”与“召回互补性”。

### 2）BM25 和向量检索是怎么融合的？
- 常见做法：分数归一化后加权融合，或先并集召回再统一排序。
- 权重可基于验证集调参，默认值可先经验设定。
- 可补充数据位：不同权重下命中率/主观准确率变化。

### 3）为什么还要 Reranker？它和向量检索有什么区别？
- 向量检索主要做“快速召回”，Reranker做“精细排序”。
- Reranker更像交叉编码器，对 query-doc 关联理解更细。
- 成本更高，因此通常只对 TopK 候选重排。

### 4）Reranker 什么时候开，什么时候关？
- 开：问题复杂、答案精度优先、可接受额外时延。
- 关：追求低时延/低成本，或数据规模较小、召回已足够稳定。
- 可以做成配置项按场景切换。

### 5）LlamaIndex 在项目中解决了什么问题？
- 统一了文档加载、切分、索引、检索、QueryEngine 组织方式。
- 降低了手写 RAG pipeline 的工程复杂度。
- 便于替换检索器、向量库、LLM 接口。

### 6）你是如何做文档切分（chunking）的？
- 切分粒度影响召回和上下文完整性。
- 常见策略：固定长度 + overlap，结合语义边界优化。
- 过大导致噪声，过小导致语义断裂；需要平衡。

### 7）开发/生产双模式存储有什么价值？
- 开发模式：快速重建索引、方便调试与实验。
- 生产模式：持久化与复用，减少启动重建时间。
- 本质是把“迭代效率”和“线上稳定性”解耦。

### 8）为什么要兼容 Ollama 和 OpenAI API？
- 本地模型适合隐私敏感和成本控制。
- OpenAI 兼容接口有更强模型生态与可用性。
- 统一适配层后可低成本切换，减少上层改动。

### 9）你如何评估 RAG 系统效果？
- 离线：检索命中、TopK 覆盖、答案相关性评估。
- 在线/主观：回答可用性、幻觉率、用户反馈。
- 可补充数据位：A/B 对比前后结果。

### 10）如何减少幻觉（Hallucination）？
- 提高检索质量：混合检索 + 重排 + 更合理 chunk。
- 提示词约束：要求“仅基于上下文回答，不确定则说明”。
- 输出中可附引用片段增强可解释性。

### 11）遇到性能瓶颈时你怎么排查？
- 分层定位：检索耗时、重排耗时、生成耗时分别测量。
- 调参方向：TopK、chunk 数、模型大小、并发策略。
- 优化策略：缓存、索引持久化、按需启用重排。

### 12）这个项目里你最核心的技术贡献是什么？
- 可答：检索策略设计（混合 + 重排）或工程封装（多模型适配 + 双模式存储）。
- 强调你做了“方案权衡与落地”，不是只调用框架。
- 最好带一个“问题->决策->结果”的闭环。

### 13）如果让你继续迭代 ThinkRAG，你会做什么？
- 增加评测集与自动化评估流程。
- 优化召回融合策略（动态权重/学习排序）。
- 增加来源引用、权限控制、监控告警等生产能力。

### 14）如果数据中有大量表格/代码块，方案会怎么调整？
- 改进解析与切分策略，保留结构信息。
- 检索时加入结构化字段过滤。
- 回答阶段优先引用结构化片段，降低误解风险。

## 9. 模拟面试（细节版：问题+参考回答）

> 说明：以下回答按当前项目代码默认参数整理，面试时可以先给“默认值”，再补“为什么这样定、什么时候调整”。

### Q1：你先完整介绍一下系统架构，不要只说用了什么框架。

**参考回答：**
我把 ThinkRAG 拆成 4 层：

1. **交互层（Streamlit）**：负责模型配置、文件/网页入库、问答和知识库管理。
2. **编排层（LlamaIndex）**：负责 ingestion pipeline、index 管理、query engine。
3. **检索生成层（Hybrid Retrieval + LLM）**：向量检索 + BM25 混合召回，支持可选 reranker，然后交给 LLM 生成答案。
4. **存储层（Dev/Prod）**：
   - Dev：本地 simple store（便于快速调试）。
   - Prod：Doc/Index/Chat 用 Redis，Vector 可切 Chroma/LanceDB/ES。

入口是 `app.py`，启动后会先初始化 `st.session_state`，再根据配置恢复当前 LLM 与检索参数；问答页面会检查索引是否存在，如果存在就创建 query engine，随后进入检索与生成流程。

---

### Q2：你们怎么做文档切分？切块长度和重叠长度是多少？为什么这么设？

**参考回答：**
我们在 `IndexManager.load_files/load_websites` 里把切分参数写到 `Settings.chunk_size` 和 `Settings.chunk_overlap`，然后交给 ingestion pipeline 执行。

当前默认值：

- `chunk_size = 2048`
- `chunk_overlap = 512`

这样设计是为了平衡三件事：

1. **上下文完整性**：2048 可以保留较完整语义段，避免 chunk 太碎导致“句子被截断”。
2. **召回精度**：如果 chunk 太大，会把无关内容一起召回，影响 rerank 和生成质量。
3. **跨块信息连续性**：512 overlap 用来保留相邻块语义连接，降低边界信息丢失。

调参策略：

- FAQ/短文本：可调小到 512~1024，降低噪声。
- 规章/长文：可维持 1536~2048，确保语义完整。
- 如果发现“答案缺半句”，优先增大 overlap；如果“命中不准”，优先减小 chunk size。

---

### Q3：为什么不是纯向量检索？BM25 在你们系统里的作用是什么？

**参考回答：**
纯向量检索对语义问题效果好，但在业务场景里，很多问题包含**专有名词、编号、短语、缩写**，这类问题 BM25 往往更稳。

所以我用了混合检索：

- 一路是 `VectorIndexRetriever`（语义召回）
- 一路是 `BM25Retriever`（关键词召回）

另外我还做了中文适配：BM25 默认 tokenizer 对中文不友好，所以我接了 `jieba` 分词。

混合后的收益是：

- 语义类问题不丢
- 关键词类问题不漏
- 在中文场景下更稳

---

### Q4：你们融合检索是怎么做的？权重和模式是什么？

**参考回答：**
我们用的是 LlamaIndex 的 `QueryFusionRetriever` 封装，当前默认是：

- `retriever_weights = [0.6, 0.4]`（向量 0.6，BM25 0.4）
- `mode = dist_based_score`
- `top_k` 默认跟系统参数一致，默认值是 `5`

为什么是 0.6/0.4：

- 语义召回仍然是主通道，所以向量权重略高。
- 同时保留 BM25 的强关键词能力，避免“语义近但关键词错位”。

如果权重设置不合理会出现两个问题：

1. BM25 权重太高：返回太“字面匹配”，上下文语义不够。
2. 向量权重太高：专有词命中下降，容易答非所问。

---

### Q5：Reranker 在你们系统怎么接入？Top N 设多少？

**参考回答：**
Reranker 是可开关的 node postprocessor：

- 默认 `use_reranker = False`（先保证速度和易用性）
- 默认 `top_n = 2`
- 默认 `reranker model = bge-reranker-base`

它的工作方式是：先从混合检索拿候选，再对候选做精排，最后把更相关的上下文喂给 LLM。

为什么默认不开：

- reranker 能提精度，但会增加时延。
- 对小知识库和简单问题，不开也能满足。

什么时候建议开：

- 复杂问题、跨段推理、答案准确率优先时。
- 候选多、噪声大时，rerank 收益更明显。

---

### Q6：问答链路从用户输入到最终答案，完整调用路径是什么？

**参考回答：**
流程是：

1. 用户在 Query 页输入问题。
2. 系统检查 LLM 是否已初始化、索引是否存在。
3. 读取当前设置（Top K、response_mode、use_reranker、top_n 等）创建 query engine。
4. query engine 内部执行混合检索（向量+BM25），可选 rerank。
5. 用 prompt 模板组装上下文，调用 LLM 生成答案。
6. 前端流式输出回答，并展示 source nodes（文件名/页码/片段/score）。

默认参数里：

- `Top K = 5`
- `Temperature = 0.1`
- `Response Mode = simple_summarize`

这些参数都可以在高级设置页动态修改并持久化。

### Q7：网页入库和文件入库有什么差异？你做了哪些容错？

**参考回答：**
文件入库是本地文件读取；网页入库多了抓取不稳定问题，所以我做了两层容错：

1. **正文有效性过滤**：过滤空文本，避免把空节点写入索引。
2. **抓取回退**：原始 URL 抓不到时，回退到 `r.jina.ai` 镜像再抓一次。

另外还做了 metadata 清洗，防止出现不可 JSON 序列化对象导致入库失败。

### Q8：开发模式和生产模式你是怎么设计的？为什么这么分？

**参考回答：**
核心是把“研发效率”和“运行稳定性”分离：

- **开发模式**：本地 simple store，零依赖，适合快速试验参数、频繁重建索引。
- **生产模式**：Doc/Index/Chat 进 Redis，向量库走 Chroma/LanceDB/ES，更适合持久化和规模化。

这样设计能让同一套业务逻辑在不同环境复用，只切存储上下文，不改核心流程。

### Q9：你怎么评估“这个 RAG 确实变好了”？

**参考回答：**
我会分三层指标看：

1. **检索层**：TopK 命中率、相关片段覆盖率。
2. **生成层**：回答正确性、完整性、可解释性（是否能从 source 复核）。
3. **性能层**：平均时延、P95、开启/关闭 reranker 的时延变化。

评估方式：

- 准备固定问题集做离线对比（纯向量 vs 混合 vs 混合+rerank）。
- 再做线上人工抽检，避免只看单一指标。

如果你面试时没有真实数字，可以说“我保留了可补充数据位，后续按评测集填写”。

### Q10：如果知识库扩大 10 倍，你预判瓶颈在哪？怎么优化？

**参考回答：**
第一瓶颈通常在检索和重排，不一定在 LLM 本身。我的优化顺序是：

1. **检索侧**：控制 TopK，分域索引，减少全库扫描。
2. **重排侧**：只对小候选集 rerank，必要时降级为不开启。
3. **存储侧**：切到更适合的向量库和索引参数，增加持久化缓存。
4. **生成侧**：选择更轻模型或压缩上下文，降低 token 开销。

目标是先把“召回质量/时延”稳定住，再优化生成成本。

### Q11：你在这个项目里最体现工程能力的点是什么？

**参考回答（STAR简版）：**

- **S/T**：系统初版可跑，但配置散、模型切换成本高、环境迁移麻烦。
- **A**：我做了统一配置与状态管理：将运行参数落到 `session_state + config_store`，并封装模型适配层（Ollama/OpenAI 兼容 API），同时做 Dev/Prod 存储分层。
- **R**：实现了“参数可追踪、模型可切换、环境可迁移”的工程闭环，后续迭代只改配置不改主链路，维护成本明显降低（可补充数据位）。

### Q12：如果让你继续做下一版，你会优先做哪三件事？

**参考回答：**
我会优先做三件事：

1. **评测体系**：建立标准问答集 + 自动回归，避免“改动后效果回退”。
2. **权限与多租户**：支持多用户多知识库隔离，满足真实业务使用。
3. **异步入库与任务监控**：把 ingestion 任务化，支持队列、重试、可观测。

这三项可以把当前 Demo 形态推进到更接近生产可用。
